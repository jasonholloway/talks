1.14 The Path to Pattern matching

1.14.1 the counterpart of callable objects

  1. transparency, not encapsulation
  2. what do you send to objects? messages!
  3. and how do you distinguish messages? by their structure

1.14.2 we always had pattern matching

  1. null checks are pattern matches
  2. branching on booleans is pattern matching!
  3. deconstructors allow more ways to distinguish inputs
    1. example of interesting pattern match of an object

1.14.3 very nearly values vs objects

  1. value types don't have identities, can be copied; objects own data or a context

    though a context can be copied

  2. objects can have multiple aspects - this is nice; they can have both interpretable shape and encapsulated behaviour
    1. as with type matching switch statements

1.14.4 interesting pattern matchings

  1. in the world of objects, we structure by inheritance
  2. polymorphic dispatch distinguishes the structure of objects for us

    we live with this all the time

    in delegating through layers, what we call encapsulation is really a way to pare away the work to be done we will interpret one level, while leaving further interpretations to something else, more specialised like a pipeline of interpretation. This passing on of the problem isn't so much of a call between two communicating objects, it is a pipeline, with the real to-and-fro happening at a layer above

1.15 Objects made transparent - the opposite of OO encapsulation

1.15.1 opaque objects that do something vs a piece of interpretable information

  1. trad C# tries to make everything like the former

1.15.2 also immutable vs mutable as second angle at this

1.15.3 instead of sending messages to opaque well-behaving thing, we carve it up and inspect it

  1. or rather, the split is between sending and receiving
  2. there's a line between commanding and offering up an interpretable request
  3. in structured programming we present and then receive
  4. pattern matching happens on receipt
  5. receipt happens in two ways, from the instigator and from the instigated

    can think of it as a table tennis match, the ball paddled here and there except simplified so the caller receives a response from the called the called knows that is its final possible communication

1.15.4 C# way is to invoke methods on objects that adhere to contracts

  1. we receive the response, and we do pattern match it already, though with a limited vocabulary of matchings
  2. Null checks are pattern matches
  3. Bool checks are pattern matches
  4. matches object/value split
  5. if/else branching is pattern matching

    the essence of computation, in C# as elsewhere: an input is provided, something is given back
